# sdp-on-prem-powerplatform

> [!WARNING]
> **Vibe-coded personal alpha — use with utmost caution.**
>
> This project was built AI-assisted ("vibe coded") for **personal use** and is in **alpha
> testing**. It is public for one reason only: Power Platform can create a custom connector

> beats re-uploading them on every test round. **Public does not mean production-ready:**

> - Expect breaking changes without notice. No support, no warranty (see [LICENSE](LICENSE)).
> - Review any definition before importing it, point it only at instances you are allowed to
>   test against, and use a **least-privilege API key** — never an admin key you care about.

> Not affiliated with, or endorsed by, ManageEngine / Zoho or Microsoft.

Microsoft **Power Platform custom connectors** for **ManageEngine ServiceDesk Plus On-Premises**
— hand-authored, lean **OpenAPI 2.0 (Swagger 2.0)** definitions covering the core Service Desk
modules, the CMDB API (v3 and legacy v1), and the Execute Query report API. Import a connector
into Power Automate / Power Apps / Copilot Studio, point it at your internet-published SDP
instance, paste an API key, and use the actions in flows.

## The connector family

| Connector | Ops | Covers | Auth |
|---|---|---|---|
| [`sdp-service-desk`](connectors/sdp-service-desk/) | 37 | Requests (+ notes, worklogs, tasks, resolution, close, delete), changes, problems, solutions, and 12 admin-lookup dropdowns | `authtoken` header |
| [`sdp-assets-cmdb-v3`](connectors/sdp-assets-cmdb-v3/) | 13 | Assets, CI types, per-type CIs, `_metadata`, association relationships (SDP build **15100+**) | `authtoken` header |
| [`sdp-cmdb-v1`](connectors/sdp-cmdb-v1/) | 4 | Legacy XML CMDB for builds **< 15100** (CI list/count, relationships) | `TECHNICIAN_KEY` query |
| [`sdp-query`](connectors/sdp-query/) | 1 | Execute Query — one read-only SQL SELECT replaces many REST GETs | `authtoken` header |

Each connector lives in `connectors/<name>/` as `apiDefinition.swagger.json` +
`apiProperties.json` + a per-connector `README.md`, plus **one "Learn more" doc per action**
under `connectors/<name>/actions/` — every parameter explained with copy-paste `input_data`
variations, linked from the imported action's About tab (`externalDocs` + a `Learn more:`
description suffix; see [ADR-0003](docs/adr/0003-per-action-learn-more-docs.md)). The three V3 connectors are generated by
`tools/gen_swagger.py` from a hand-curated operation table (see [ADR-0001](docs/adr/0001-initial-decisions.md));
the V1 connector is hand-authored. **The committed JSON is the reviewed artifact** — this is not
a mechanical conversion of the official Postman collection (which is ~9 MB, over Power Platform's
1 MB import cap).

## Install a connector

1. **Generate an API key in SDP.** *Admin → Technicians*, edit a technician (ideally a
   least-privilege integration account), **Generate API Key**, set a validity, copy it.
   - For `sdp-query`, that technician also needs the **Create Query Report** permission.
2. **Import.** Power Automate / Apps → *Custom connectors → New custom connector*, then either
   **Import an OpenAPI file** (a downloaded `apiDefinition.swagger.json`) or import from URL
   with the raw URL of the connector you want:
   - `https://raw.githubusercontent.com/PBNZ/sdp-on-prem-powerplatform/main/connectors/sdp-service-desk/apiDefinition.swagger.json`
   - `https://raw.githubusercontent.com/PBNZ/sdp-on-prem-powerplatform/main/connectors/sdp-assets-cmdb-v3/apiDefinition.swagger.json`
   - `https://raw.githubusercontent.com/PBNZ/sdp-on-prem-powerplatform/main/connectors/sdp-cmdb-v1/apiDefinition.swagger.json`
   - `https://raw.githubusercontent.com/PBNZ/sdp-on-prem-powerplatform/main/connectors/sdp-query/apiDefinition.swagger.json`

   Use these `raw.githubusercontent.com` URLs exactly — the `github.com/…/blob/…` page URL
   serves HTML, and the portal fails with *"We weren't able to download the OpenAPI file"*.
3. **Set the host.** Edit the connector's **Host** from the placeholder `sdp.example.com` to your
   SDP server (e.g. `servicedesk.company.com`). **HTTPS only.**
4. **Create a connection.** Paste the API key when prompted — it is stored as a **secured**
   parameter (V3 connectors send it as the `authtoken` header; the V1 connector as the
   `TECHNICIAN_KEY` query parameter). It is never stored in the definition files.

## Networking — direct vs on-premises gateway

- **Default: direct HTTPS** to your internet-published SDP host (the common shape, like the demo).
- **Firewalled installs: the on-premises data gateway.** Every connector declares the `gateway`
  capability in `apiProperties.json`, so the gateway option appears at connection time — **no
  change to the definition**, it's a per-connection runtime choice.
- Swagger 2.0 allows one `host`, so each customer edits the placeholder host on import. (A
  dynamic-host policy that serves any host from one connector is a possible future enhancement.)

## Key API facts baked in

- **`input_data` envelope.** SDP rejects any parameter not wrapped in a single `input_data`
  value (JSON for V3, XML for V1). Each operation therefore takes one **Input Data** parameter
  carrying the envelope, with a copy-paste example and sensible default. (Friendly typed
  parameters that assemble `input_data` are a future enhancement for top actions.)
- **Pagination** (V3): `list_info` — `row_count` (max 100), `start_index`, `has_more_rows`,
  `get_total_count`, plus `search_criteria` and `sort_field`/`sort_order`.
- **Responses** (V3): `{ response_status, <module>: ..., list_info }`; success is
  `response_status.status_code` **2000**. `response_status` is an **array** on list responses
  and an **object** on single-resource and error responses in every capture so far — the
  response schemas leave it untyped so consumers handle both shapes.

## Build-version caveats

- **CMDB v3 needs SDP build 15100+** (or AssetExplorer 7700+). Below that, use `sdp-cmdb-v1`.
  Check *Admin → About / build number*.
- **Execute Query is admin-scoped and edition/build-gated.** Probe with `SELECT 1 AS test`
  (`templates/00-capability-probe.sql`) at setup; fall back to REST if it errors.
- **Lookup endpoints vary:** `sdp-service-desk` points *List requesters* at `/users` and
  *List support groups* at `/support_groups` (verified on the demo build); some builds use
  `/requesters` and `/groups` instead — edit the operation path if a lookup 404s.

## Execute Query & schema fragility

The `sdp-query` connector runs one read-only SQL `SELECT` (JOINs, custom fields, aggregates) to
replace many REST GETs. Physical table/column names are **build-specific**; the [`templates/`](templates/)
library is keyed to **build 14990** and every template was verified to run live. After an SDP
upgrade, names can change — keep SQL in editable flow variables / template files, not hard-baked,
and regenerate the schema map with `pg_dump --schema-only` against your instance's Postgres.

## Verification status

- **Reads:** every read operation across all four connectors returns **2xx / status_code 2000**
  live against `https://demo.servicedeskplus.com` (read-only) — captured in
  [`docs/test-evidence/`](docs/test-evidence/).
- **Execute Query:** SELECT returns rows; `UPDATE`/`information_schema`/`GET` are rejected — all
  verified live.
- **Writes** (create/update/delete/close, CI writes, relationship writes): authored and
  structurally verified (valid Swagger 2.0, fresh-context reviewed), but **not yet verified
  live** — a write 2xx is pending running the smoke test with a technician API key on a
  disposable instance (see [`docs/deploy-sdp-wsl2.md`](docs/deploy-sdp-wsl2.md) and
  `tools/live-test.ps1 -IncludeCreate`). **This is the main reason for the alpha warning
  above.** Never write to the shared demo.
- Each connector was reviewed by a fresh-context verifier subagent against the Power Platform
  rules and the SDP API contract.
